## Custom Chat Web UI — Implementation & Deployment

### What was done
- Created `web/serve.js`: Zero-dependency Node.js server (~55 lines)
  - Serves `index.html` for all HTTP requests on :18790
  - Proxies WebSocket `/ws` to OpenClaw gateway at :18789 via raw TCP pipe
  - Rewrites Origin and Host headers to pass gateway allowedOrigins check
  - Injects `GATEWAY_TOKEN` env var into HTML at serve time
- Created `web/index.html`: Single-file chat UI (~400 lines HTML+CSS+JS)
  - Google-style centered landing page with about watermark text
  - Transitions to chat mode after first message (fade animation)
  - Color palette: silver/white primary, earth tones secondary, deep navy accent
  - OpenClaw WebSocket protocol (reverse-engineered from bundle):
    - Wait for `connect.challenge` event
    - Send `{type:"req"}` with UUID id, auth token, webchat client, operator scopes
    - Extract sessionKey from `payload.snapshot.sessionDefaults.mainSessionKey`
    - `chat.history` for previous messages
    - `chat.send` with idempotencyKey
    - Stream via `event:"chat"` with `payload.state: "delta"|"final"`
    - Message content: array of `{type:"text",text}` blocks
  - Strips `<think>...</think>` tags from Qwen3.5 output
  - XSS prevention via escapeHtml()
  - Auto-reconnect with exponential backoff
  - Basic markdown rendering (code blocks, bold, italic, links)
- Created `deploy/marcos-chat-web.service`: Systemd unit with GATEWAY_TOKEN
- Deployed to marcos, installed systemd service, updated Tailscale serve
- **E2E test passed**: "who are you?" → streaming response from Qwen3.5

### Protocol discoveries (not documented anywhere)
- Request frame `id` must be a non-empty string (UUID), not a number
- `client.id` must match a known constant (e.g., `"webchat"`)
- Gateway requires `auth.token` when `gateway.auth.mode` is `"token"`
- Scopes `["operator.admin","operator.write","operator.read"]` needed for chat.send
- Origin header must match `gateway.controlUi.allowedOrigins`
