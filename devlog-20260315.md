### What was done

2. Installed `cloudflared 2026.3.0` on Jetson via Cloudflare apt repo
3. Created Cloudflare Tunnel `marcos` (ID: `d42cf58b-19c5-4a0a-8a93-02b364a480b9`)
4. Tunnel registered and connected to Cloudflare edge at `lax05` via http2
5. Changed nameservers from DNSPod to Cloudflare (`agustin.ns.cloudflare.com`, `anita.ns.cloudflare.com`)
6. Waiting for Cloudflare to verify nameserver change

### Issues encountered & fixes
- **Jetson can't reach GitHub**: Used Cloudflare's apt repo instead
- **SRV DNS lookup failure**: Router DNS (192.168.3.1) can't handle SRV records. Fixed with `edge: [198.41.192.7:7844]` in config
- **Systemd startup timeout**: Default 15s too short. Increased to 60s
- **QUIC protocol timeout**: Switched to `protocol: http2`
- **CNAME from DNSPod doesn't work**: `cfargotunnel.com` only works with Cloudflare's DNS proxy. Moved nameservers to Cloudflare

### Config files on Jetson
- `/etc/cloudflared/config.yml` — tunnel config
- `/etc/systemd/system/cloudflared.service` — systemd unit (TimeoutStartSec=60)
- `/home/nvidia/.cloudflared/cert.pem` — Cloudflare auth cert
- `/home/nvidia/.cloudflared/d42cf58b-19c5-4a0a-8a93-02b364a480b9.json` — tunnel credentials

### Architecture
```
Internet → marcos.chat (Cloudflare DNS)
  → Cloudflare Edge (lax05, port 7844)
  → Cloudflare Tunnel (http2)
  → Jetson cloudflared → localhost:18789 (OpenClaw)
```

### Resolution
- [x] Domain live and verified from phone, Mac browser, and curl
- [x] End-to-end working: browser → marcos.chat → OpenClaw webchat

### Additional issues & fixes (continued debugging)
- **Mac browser couldn't access marcos.chat**: Root cause was **ClashX Pro** (proxy on port 7890) in TUN/Enhanced Mode routing ALL traffic through a foreign proxy. The proxy path hit a Cloudflare edge that dropped marcos.chat TLS connections.
  - Fix: Added `marcos.chat` and `*.marcos.chat` to macOS proxy bypass list (`networksetup -setproxybypassdomains Wi-Fi`)
- **Tailscale MagicDNS interference**: Disabled MagicDNS and set `--accept-dns=false` during debugging, then restored both after identifying ClashX as the real cause
- **SSH daemon lockup**: Too many SSH sessions during debugging caused MaxStartups exhaustion. Recovered after waiting and using Tailscale IP

### Lessons learned
1. ClashX Pro TUN mode captures ALL traffic at the network level — proxy bypass settings still work for browser traffic
2. Tailscale's `utun` interfaces can coexist with ClashX Pro's `utun` — they're separate tunnels
3. When debugging network issues on Mac, always check `networksetup -getwebproxy Wi-Fi` for system proxy settings
4. Cloudflare Tunnel needs `protocol: http2` and explicit `edge` IPs when router DNS can't resolve SRV records
5. `cloudflared` systemd service needs `TimeoutStartSec=60` (default 15s is too short)
